# Boundary Conditions

Boundary conditions (BCs) in FUSS are specified through a combination of **per-block face assignments** in `input.ini` and **named BC sections** that hold the parameters of each BC. A face-by-face boundary-condition table (`INPUT/bc.txt`) is then produced from the INI assignments (or generated independently by ATLAS) and read by the solver.

!!! info "BC files and ATLAS"
    In the normal Hydra workflow, the `bc.txt` file is generated by the **ATLAS** pre-processor, which translates the INI-level keyword BCs into the internal numeric codes that the solver consumes. If ATLAS is not available, `bc.txt` can be prepared manually as long as it conforms to the format described in the [Internal `bc.txt` format](#internal-bctxt-format) section below.

---

## Face Numbering

For a block with $n_i \times n_j \times n_k$ cells the six faces are labelled following the ORION convention:

| Face index | Location | Varying indices |
|:----------:|----------|-----------------|
| 1 | $i = 1$ (imin, $-x$) | $j = 1 \ldots n_j$, $k = 1 \ldots n_k$ |
| 2 | $i = n_i$ (imax, $+x$) | $j = 1 \ldots n_j$, $k = 1 \ldots n_k$ |
| 3 | $j = 1$ (jmin, $-y$) | $i = 1 \ldots n_i$, $k = 1 \ldots n_k$ |
| 4 | $j = n_j$ (jmax, $+y$) | $i = 1 \ldots n_i$, $k = 1 \ldots n_k$ |
| 5 | $k = 1$ (kmin, $-z$) | $i = 1 \ldots n_i$, $j = 1 \ldots n_j$ |
| 6 | $k = n_k$ (kmax, $+z$) | $i = 1 \ldots n_i$, $j = 1 \ldots n_j$ |

For 2-D plane cases (`nk = 1`), faces 5 and 6 are degenerate and should be set to `null`.

---

## Per-Block Face Assignment

Boundary conditions are assigned per block face using `[BCB-Block*]` sections. Each face references a named BC section defined elsewhere in `input.ini`, or one of the built-in keywords `null` and `connection`:

```ini
[BCB-Block1]
face1 = Twall-cold
face2 = connection
face3 = adiabatic
face4 = convection
face5 = null
face6 = null
```

| Keyword | Meaning |
|---------|---------|
| `null` | Degenerate / inactive face — no BC applied (typical for face 5/6 on 2-D plane cases) |
| `connection` | Block-to-block interface (continuity of $T$ and $k\nabla T\!\cdot\!\hat{\mathbf n}$); the matching face on the neighbour block is inferred from the mesh layout |
| `<name>` | Any other identifier — references a `[<name>]` section that defines a wall-type BC by parameter combination (see below) |

The same named section can be referenced by several faces or several blocks, which avoids duplicating identical BC configurations.

---

## Wall-Type BCs by Parameter Combination

All wall BCs in the INI file use `type = wall`. The **physical behaviour is selected by which parameters are present** in the section. The pre-processor (ATLAS or the built-in INI translation) maps each parameter combination to one of the internal numeric codes consumed by the solver.

| Parameter combination in `[<name>]` | Physical behaviour | Internal code |
|-------------------------------------|--------------------|:-------------:|
| `q = <value>` *(or `q-time = file`)* | Prescribed heat flux $q_w$ (W/m²); set `q = 0` for an adiabatic face | `301` |
| `T = <value>` *(or `T-time = file`)* | Prescribed wall temperature $T_w$ (Dirichlet) — constant or time-varying | `302` |
| `hconv`, `Tref`, `qrad` | Convective heat transfer with an optional radiative pre-load $q_w$ | `303` |
| `eps`, `Tref` | Pure radiation (Stefan–Boltzmann) | `304` |
| `hconv`, `eps`, `Tref` | Combined convection + radiation | `305` |

!!! warning "Choose one parameter family per section"
    Each named section is interpreted by the parameters it carries. Mixing keys from different families in the same section (for example `T` together with `hconv`) is not allowed and leads to an invalid translation.

### Parameter reference

| Parameter | Units | Description |
|-----------|-------|-------------|
| `T` | K | Constant wall temperature |
| `T-time` | filename | Two-column ASCII table `time(s) T(K)`; optional `periodic` header repeats cyclically; linear interpolation between entries |
| `q` | W/m² | Constant prescribed normal heat flux (positive = into the solid) |
| `q-time` | filename | Two-column ASCII table `time(s) q(W/m²)` |
| `hconv` | W/(m²·K) | Convective heat-transfer coefficient |
| `Tref` | K | Reference temperature for convection / radiation |
| `eps` | — | Surface emissivity, $0 < \varepsilon \le 1$ |
| `qrad` | W/m² | Radiative pre-load (set `0` for pure convection) |

!!! warning "Radiation stability"
    Pure-radiation (`eps`+`Tref`) and combined convection+radiation (`hconv`+`eps`+`Tref`) BCs introduce a quartic nonlinearity in $T$. When either is active, smaller VNN values (typically 0.5 or below) are recommended to maintain stability of the explicit time-stepping.

## BC Section Examples

**Adiabatic (zero heat flux):**

```ini
[adiabatic]
type = wall
q    = 0.0
```

**Prescribed heat flux:**

```ini
[heat-flux]
type = wall
q    = 1.0e5         ; W/m² into the solid
```

**Prescribed temperature (constant):**

```ini
[Twall-hot]
type = wall
T    = 2000.0
```

**Prescribed temperature (time-varying):**

```ini
[Tw_variable]
type   = wall
T-time = Tw_time.dat
```

`Tw_time.dat` is a two-column ASCII file: `time(s)  T(K)`. The file can optionally begin with the keyword `periodic`, in which case the table is repeated cyclically for the entire simulation. Linear interpolation is used between table entries.

**Convection:**

```ini
[convection]
type  = wall
hconv = 750.0        ; W/(m²·K)
Tref  = 273.15       ; K
qrad  = 0.0          ; W/m² — set 0 for pure convection
```

**Radiation:**

```ini
[radiation]
type = wall
eps  = 0.98          ; emissivity, 0 < ε ≤ 1
Tref = 300.0         ; K (radiation-sink temperature)
```

**Combined convection + radiation:**

```ini
[mixed-loss]
type  = wall
hconv = 25.0         ; W/(m²·K)
eps   = 0.85
Tref  = 300.0        ; K
```

**Built-in keywords — no section required:**

```ini
[BCB-Block1]
face1 = null         ; degenerate face (2-D z-direction)
face2 = connection   ; block-to-block interface
```

## Internal `bc.txt` format

`INPUT/bc.txt` is the raw, face-by-face boundary condition file read directly by the solver. It is the ground truth from which the solver builds its BC data structures. For simple cases it is generated automatically from the `[BCB-Block*]` assignments in `input.ini`; for complex multi-block geometries it is typically produced by ATLAS.

The file is structured as a list of entries, one per boundary cell-face:

```
<block>  <i>  <j>  <k>  <face>  <type>
<data line (if required by type)>
```

**Line 1 — face identifier:**

| Field | Description |
|-------|-------------|
| `block` | 1-based block index |
| `i`, `j`, `k` | Cell indices of the boundary cell (1-based) |
| `face` | Face index of that cell (1–6, see [Face Numbering](#face-numbering)) |
| `type` | Numeric BC type code (see table below) |

**Line 2 — BC data (type-dependent):**

The format and presence of the data line depend on the numeric `type` code. Internal codes used by the solver are:

| Code | Behaviour | Data line |
|:----:|-----------|-----------|
| `101` | Block connection | `bs is js ks fs  d11 d12 d21 d22` (target block/face + index direction cosines) |
| `102` | Chimera overset | `n1 n2` then `n1+n2` donor lines: `bD iD jD kD vol_frac` |
| `103` | Multi-solver coupling | `bs is js ks fs  d11 d12 d21 d22` |
| `300` | Symmetry plane | — *(no data line)* |
| `301` | Prescribed heat flux | `qw` or filename for time-varying $q_w(t)$ |
| `302` | Prescribed temperature | `Tw` or filename for time-varying $T_w(t)$ |
| `303` | Convection (+ radiative pre-load) | `hconv, qrad, Tref` |
| `304` | Pure radiation | `eps, Tref` |
| `305` | Convection + radiation | `hconv, eps, Tref` |

## Examples

**1-D rod (100 cells along $x$, hot left face, adiabatic transverse faces)**

`input.ini`:

```ini
[BCB-Block1]
face1 = Twall-hot
face2 = Twall-cold
face3 = adiabatic
face4 = adiabatic
face5 = null
face6 = null

[Twall-hot]
type = wall
T    = 2000.0

[Twall-cold]
type = wall
T    = 1000.0

[adiabatic]
type = wall
q    = 0.0
```

Generated `bc.txt`:

```
       1       1       1       1       1     302
    0.200000E+04
       1     100       1       1       2     302
    0.100000E+04
       1       1       1       1       3     301
    0.000000E+00
       1       2       1       1       3     301
    0.000000E+00
       ...
       1     100       1       1       4     301
    0.000000E+00
```

Faces 1 and 2 each have one entry (the single cell at each end of the 1-D domain) and carry Dirichlet temperatures (code 302). Faces 3 and 4 each have 100 entries (one per $i$-cell) and are adiabatic (code 301, $q_w = 0$).

**Block connection (type 101)**

`input.ini`:

```ini
[BCB-Block1]
face2 = connection
```

Generated `bc.txt`:

```
       1       1      16       1       4     101
       2       1       1       1       3       1       0       0       1
```

The data line carries the target block and face information plus the direction cosines.

**Convective face**

`input.ini`:

```ini
[convection]
type  = wall
hconv = 750.0
Tref  = 273.15
qrad  = 0.0
```

Generated `bc.txt` line for one cell-face:

```
       1       1      16       1       4     303
    0.750000E+03,    0.000000E+00,    0.273150E+03
```
