# Running FUSS

This page describes the complete workflow for running a FUSS simulation: preparing the case directory, defining the needed files, launching the solver, and inspecting the output.

All configuration is driven by:
- `input.ini` — the main INI file that defines simulation parameters, including the numerical settings, mesh generation, initial state, and boundary-condition assignments.
- `INPUT/` — a directory containing the supporting data files for the simulation, such as the material property table, boundary-condition table, and any time-varying BC tables.

!!! tip
    All files within `INPUT/` can be built using the FUSS pre-processing tools provided by ATLAS. ATLAS is a very powerful tool, but it is not required to run FUSS. All files can be created manually or with custom scripts as long as they follow the expected format. For simple Cartesian geometries, FUSS also exposes a built-in mesh generator (GRIB) that runs automatically when the mesh files are absent.

For the detailed format of material, initial, and boundary condition data see the dedicated reference pages:

- [Input File](input.md)
- [Initial Conditions](initial-conditions.md)
- [Boundary Conditions](boundary-conditions.md)

## Simulation Workflow

A typical FUSS run follows the following phases:

1. **Setup** — define the mesh, material properties, initial and boundary conditions
2. **Preprocess** — run ATLAS (or the built-in GRIB mesh generator) to produce the supporting files in `INPUT/` and `MESH/`
3. **Solve** — advance the temperature field in pseudo-time or real time (explicit Runge–Kutta)
4. **Postprocess** — write solution fields, wall data, probe time-histories, and diagnostics

All configuration is driven by the INI file (`input.ini`).

## Case Directory Structure

Every FUSS case follows a standard layout:

```
my_case/
├── input.ini                 ← case configuration
├── FUSS.sh                   ← convenience run script (optional)
├── INPUT/
│   ├── phase.txt             ← material phase definition
│   ├── properties.dat        ← tabulated material properties
│   └── bc.txt                ← face-by-face boundary condition data
├── MESH/
│   └── mesh.tec              ← computational grid (auto-generated by GRIB if absent)
└── SOLUTION/                 ← created at runtime
    ├── field.*               ← solution fields (when sol-overwrite = true)
    ├── field1.*              ← solution snapshots (when sol-overwrite = false)
    ├── field2.*
    ├── ...
    ├── wall.*                ← wall quantities (heat fluxes, surface temperatures)
    ├── residual-history.dat  ← residual norm vs. iteration
    └── <name>.tec            ← time-history at probe locations
```

!!! warning
    FUSS reads the grid geometry from `MESH/mesh.tec`. If this file is missing and a `[GRIB-meshgen]` section is present in `input.ini`, the mesh is generated automatically at startup. Otherwise the solver aborts with a loading error.

## Running the Solver

Every simulation can be launched with the following steps:

```bash
export OMP_NUM_THREADS=4
ulimit -s unlimited
./bin/FUSS
```

!!! warning
    FUSS may require a large stack size. Always run `ulimit -s unlimited` (or set `export KMP_STACKSIZE=100M` for Intel compilers) before launching the solver.

!!! tip
    A basic script like `FUSS.sh` is recommended to ensure the simulation is launched smoothly. The bundled test cases all carry a `FUSS.sh` helper that wraps `OMP_NUM_THREADS`, MPI launch, and the executable.

**Common launch patterns via `FUSS.sh`:**

```bash
./FUSS.sh solve                  # serial
./FUSS.sh -p 4 solve             # OpenMP, 4 threads
./FUSS.sh -m 2 solve             # MPI, 2 ranks
./FUSS.sh -m 2 -p 10 solve       # MPI × OpenMP: 2 ranks × 10 threads
./FUSS.sh -b solve               # background execution
```

## Logging and Diagnostics

FUSS prints diagnostic information to the console during loading and iteration. This includes:

- Loading status of input files (material properties, initial and boundary conditions)
- Summary of the case set-up (number of blocks, cells, boundary faces, types of boundary conditions, material model details, numerical scheme choices)
- Iteration count and energy residual during the run

!!! tip "Loading checks"

## Output

After a run completes, the `SOLUTION/` directory contains:

| File | Description |
|------|-------------|
| `field.*` | Solution fields ($T$, material ID, optional diagnostics) |
| `residual-history.dat` | Iteration-by-iteration residual norm |
| `wall.tec` | Wall heat-flux and surface-temperature data |
| `<probe>.tec` | Time-history data at probe locations |

### Input/Output Formats

Controlled by `sol-format` in `[FUSS-IO]`:

| Value | Format |
|-------|--------|
| `tecplot ascii` | Tecplot ASCII (`.tec`) — default |
| `tecplot binary` | Tecplot binary (`.szplt`) — requires TecIO |
| `vtk ascii` | VTK ASCII (`.vts`) |
| `vtk raw` | VTK binary (`.vts`) |

### Output Frequency

Controlled by `[FUSS-IO]` in `input.ini`:

| Parameter | Description |
|-----------|-------------|
| `sol-diter` | Write solution every N iterations (steady-state) |
| `sol-dtime` | Write solution every N seconds (time-accurate) |
| `sol-overwrite` | If `true`, overwrite the solution file each time (saves disk space) |

## Probes

Point-measurement time histories are configured with probe sections:

```ini
[FUSS-Probes]
probe1 = p1
probe2 = p2

[p1]
index-position = 1 90 1 1
variables = T
dtime = 1.0e-1
```

| Parameter | Description |
|-----------|-------------|
| `index-position` | `block i j k` — cell index of the probe location |
| `variables` | Variables to record (currently `T`) |
| `dtime` | Output time interval (s) |
| `diter` | Output iteration interval |

Each probe writes a text file `SOLUTION/p1.tec` (or `.txt`) with columns for time and the requested variables.

---

## Restart

To continue a simulation from a previous run, set `newrun = false` in `[FUSS-Parameters]`. FUSS then reads the latest solution from `SOLUTION/` and resumes integration from that point. The termination criteria (`res-threshold`, `time-threshold`, `iter-threshold`) can be updated before restarting.

See [Initial Conditions](initial-conditions.md#restart) for the underlying mechanism.

## Using FUSS as a Library

FUSS can be linked as a static library (`libFUSSL.a`) from an external Fortran program. The main entry point is the `FUSS_type` derived type:

```fortran
use FUSS_Procedures_m

type(FUSS_type) :: solver

call solver%setup(simulation)
call solver%solve(simulation)
call solver%postprocess(simulation)
```

Link against `libFUSSL.a` as described in the [Installation](../getting-started/installation.md#library-linking-advanced) page.
